# Launch Checklist

This repo is the canonical public home for the `ship` skill family.

## Day 0 Repo Setup

1. Create `ragnos-labs/agent-skills` as a public GitHub repository.
2. Set the repo description to a short GitHub-first message.
3. Set the homepage to the repo URL on launch day. If `ragnos.io/skills` exists later, keep it as a thin mirror back to the repo.
4. Disable unused surfaces if they are not part of the launch plan.

Recommended settings:

- Issues: on
- Discussions: on
- Wiki: off
- Projects: off

## Topics

Add these starter topics:

- `agent-skills`
- `ai-agents`
- `codex`
- `claude-code`
- `developer-tools`
- `release-automation`
- `open-source`
- `github`

## Discussions

Enable Discussions and ensure these categories exist:

- Q&A
- Ideas
- Requests
- Show and Tell

Seed at least these threads:

1. `Welcome to RAGnos Labs Agent Skills`
   - Pin it.
   - Explain what the repo is for, what is in scope, and where to ask questions.
2. `Requests: what should we publicize next?`
   - Keep this unpinned or secondarily pinned.
   - Ask for workflow pain points, adapter gaps, and example repos.

## Starter Issues

Create at least three starter issues:

1. `Docs: add Windows-friendly install and copy-mode examples`
   - Labels: `documentation`, `good first issue`
2. `Examples: add a sample hook map for npm plus pytest repos`
   - Labels: `enhancement`, `help wanted`
3. `Validation: add release-note checks for install examples`
   - Labels: `enhancement`

## First Release

1. Tag the initial public release as `v0.1.0`.
2. Publish release notes with exact install examples for Codex and Claude Code.
3. Link the release in launch posts and in the mirror page if one exists.

## Launch Posts

The launch content sequence is:

1. teaser,
2. Codex demo,
3. Claude demo,
4. what stayed private vs public,
5. request for issues, examples, and edge cases.

The primary call to action is always:

- star,
- watch,
- follow,
- discuss,
- contribute.

## Weekly Review

Review these GitHub-native signals each week:

- stars,
- watchers,
- forks,
- installs inferred from release traffic and feedback,
- issue volume,
- Discussion volume,
- outside pull requests.
